# SOLID 원칙
SOLID 원칙이란 객체지향 설계에서 지켜줘야 할 5개의 소프트웨어 개발 원칙을 말한다.

- SRP(Single Responsibility Principle): 단일 책임 원칙
- OCP(Open Closed Priciple): 개방 폐쇄 원칙
- LSP(Listov Substitution Priciple): 리스코프 치환 원칙
- ISP(Interface Segregation Principle): 인터페이스 분리 원칙
- DIP(Dependency Inversion Principle): 의존 역전 원칙

좋은 설계란 시스템에 새로운 요구사항이나 변경사항이 있을 때, 영향을 받는 범위가 적은 구조를 말한다. 설계가 좋다면 시스템에 예상하지 못한 변경사항이 발생하더라도, 유연하게 대처하고 이후에 확장성이 있는 시스템을 만들 수 있다.

이것은 코드를 확장하고 유지 보수 관리하기가 더 쉬어지며, 불필요한 복잡성을 제거하여 리팩토링에 소요되는 시간을 줄임으로써 프로젝트 개발의 생산성을 높일 수 있다.

## SRP (Single Responsibility Principle) 단일 책임 원칙
- 클래스는 단 하나의 책임을 가져야 한다.
- 하나의 클래스는 하나의 기능을 담당하여 하나의 책임을 수행하는데 클래스를 여러개 설계한다.
- 이것이 지켜지지 않는다면 클래스에 여러개의 기능이 있다면 기능 변경이 일어났을 때 수정해야할 코드가 많아진다. 해당 원칙을 지켜 한 책임의 변경이 다른 책임의 변경으로 이어지는 연쇄작용을 막을 수 있다.
- 그러나 책임의 범위는 추상적이기 때문에 100% 정해진 것이 없다. 그래서 어떤 프로그램을 개발하느냐에 따라 개발자마다 기준이 달라질 수 있다.
- 확실한 것은 응집도는 높고 결합도는 낮은 프로그램을 설계하는 것이 SRP의 본질이며 객체지향 설계의 핵심이라 생각한다.

<br>

## OCP (Open Closed Priciple) 개방 폐쇄 원칙
- 확장에는 열려있어야 하고, 변경에는 닫혀 있어야 한다.
- 기존의 코드를 변경하지 않고 추가할 수 있으며 수정은 최소화하도록 프로그램을 설계하는 것이다.
- 추상화와 상속등을 통해 구현할 수 있으며 자주 변화하는 부분을 추상화함으로써 기존 코드를 수정하지 않고도 기능을 확장할 수 있도록 함으로써 유연함을 높이는 것이 핵심이다.

<br>

## LSP (Listov Substitution Priciple) 리스코프 치환 원칙
- 하위 타입 객체는 상위 타입 객체에서 가능한 행위를 수행할 수 있어야 한다.
- 다형성의 원리를 이용하기 위한 원칙으로 볼 수 있다. 이것은 즉 상속관계가 아닌 클래스들을 상속관계로 설정하면 안된다는 것을 의미한다.
- 리스코프 원칙을 지키지 않으면 개방 폐쇄 원칙을 위반하게 되는 것이다. 상속 관계가 아닌 것을 수정하기 위해 기존의 코드를 여러 번 수정해야 하기 때문이다.

<br>

## ISP (Interface Segregation) 인터페이스 분리 원칙
- 어떤 클래스를 참조해야하는 상황이 생긴다면 직접 참조하는 것이 아니라 그 대상의 추상 클래스나 인터페이스로 참조하는 원칙
- 구현 클래스에 의존하지 말고, 인터페이스에 의존하라는 의미이다.
- 각 클라이언트가 필요로 하는 인터페이스들을 분리함으로써, 클라이언트가 사용하지 않는 인터페이스에 변경이 발생하더라도 영향을 받지 않도록 해야한다.
- 의존 관계를 맺을 때 변화하기 쉬운 것 또는 자주 변화하는 것보다는, 거의 변화가 없는 것에 의존하는 것
- 클래스 간의 결합도를 낮추는 것이다.

<br>

## 결론
SOLID 원칙을 준수하여 구현 클래스에 의존하지 않고 추상 클래스에 의존함으로써 유연하고 확장성 있는 애플리케이션을 만드는 것이다.

그렇게 된다면 코드는 자연스럽게 단순하게 정의될 것입니다. 단순하게 작성된 코드는 클린코드가 될 것이며 이것은 언제나 누군가가 봤을 때 코드를 봤을 때 주의 깊고 여러 사항들을 고민하였음을 느낄 수 있게 할 것입니다.
